# SpecMon: Detecting Unreliable Verifier Feedback via Specification Monotonicity in Code Generation

## Motivation

Existing code generation approaches that use formal verification (e.g., 'From Natural Language to Verified Code' and 'Clover') iteratively refine code based on verifier feedback, implicitly assuming the verifier is perfect. However, verifiers can be incomplete or buggy, causing the verification loop to converge to incorrect or unsatisfiable specifications. The root cause is the lack of an internal model of verifier reliability, making the system vulnerable to oracle failures.

## Key Insight

The verifier's output obeys a logical monotonicity property across related specifications—if a stronger specification is satisfied, then any weaker specification must also be satisfied—so violations of monotonicity reveal when the verifier's feedback is unreliable.

## Method

### Method (A) What it is
SpecMon is a consistency-checking module that takes as input a problem description, an LLM (e.g., GPT-4), and a verifier (e.g., Dafny's verifier). It outputs a reliability flag for the verifier's feedback on a candidate code-spec pair.

### (B) How it works
Below is pseudocode showing the core operation.

```python
def spec_mon(problem_desc, llm, verifier, k=3, tau=0.5):
    # Step 1: Generate candidate code and k specification pairs
    code = llm.generate_code(problem_desc)
    spec_pairs = []
    for i in range(k):
        # Instruct LLM to generate a pair (spec_strong, spec_weak) such that spec_strong logically implies spec_weak
        spec_strong, spec_weak = llm.generate_spec_pair(problem_desc, relation="stronger->weaker")
        # Step 1a: Verify logical implication using Z3 (check that spec_strong ⇒ spec_weak is valid)
        if not z3.check_implication(spec_strong, spec_weak):
            # If implication fails, discard this pair and retry up to some limit
            continue
        spec_pairs.append((spec_strong, spec_weak))
    
    # If fewer than floor(k/2) valid pairs remain, flag as "unreliable" due to insufficient evidence
    if len(spec_pairs) < k // 2:
        return "unreliable"
    
    # Step 2: Check monotonicity for each pair
    inconsistencies = []
    for spec_strong, spec_weak in spec_pairs:
        result_strong = verifier.verify(code, spec_strong)  # returns accept or reject
        result_weak = verifier.verify(code, spec_weak)
        if not is_monotonic(result_strong, result_weak):
            inconsistencies.append((spec_strong, spec_weak))
    
    # Step 3: Output reliability flag
    if len(inconsistencies) / len(spec_pairs) > tau:
        return "unreliable"
    else:
        return "reliable"

def is_monotonic(r_strong, r_weak):
    # Monotonicity: accept_strong -> accept_weak, and reject_weak -> reject_strong
    if r_strong == "accept" and r_weak != "accept":
        return False
    if r_weak == "reject" and r_strong != "reject":
        return False
    return True
```

### (C) Why this design
We chose to generate specification pairs via LLM rather than using a fixed library (e.g., pre-defined templates) because the logical structure must be context-dependent; a fixed set would miss problem-specific invariants. We chose monotonicity over other consistency checks such as consistency across paraphrased specifications (e.g., same logical content in different wording) because monotonicity is a necessary condition for any sound verifier, making violations diagnostic of verifier error rather than semantic ambiguity. We chose to flag unreliability via a threshold on the fraction of inconsistent pairs rather than requiring all pairs to be consistent, to tolerate occasional LLM generation errors; the cost is that we may miss subtle verifier failures when the threshold is too lenient. We chose not to automatically adjust the spec or code based on inconsistency, because doing so would risk masking the verifier's failure; instead, we signal the user or fallback to a simpler spec, which may reduce automation but preserves trustworthiness. **Crucially, we added a preliminary Z3 verification step to ensure that only logically correct spec pairs enter the monotonicity check, thereby grounding the load-bearing assumption that spec_strong ⇒ spec_weak holds.**

### (D) Why it measures what we claim
The computational quantity `is_monotonic(result_strong, result_weak)` measures **verifier reliability** because a fundamentally sound verifier must be monotonic with respect to logically ordered specifications—if it is not, the verifier has a bug or incompleteness. This assumption fails when the specification pair generated by the LLM is not actually logically ordered (e.g., due to ambiguous or incorrect specification generation); in that case, a monotonicity violation reflects LLM error rather than verifier unreliability. To mitigate this, we now verify the implication with Z3, so that only correctly ordered pairs are used. The second quantity, the ratio of inconsistent pairs `len(inconsistencies)/len(spec_pairs)`, measures **aggregate verifier unreliability** because random verifier errors would affect a fraction of pairs, while a systematic problem would affect all. This assumption fails when LLM generation errors are systematic across pairs (e.g., all pairs have the same misordering), but with Z3 verification we ensure the order is correct, so the ratio faithfully reflects verifier inconsistency.

## Contribution

(1) A novel consistency-checking method for formal verifiers that does not require an external oracle, using specification monotonicity. (2) The insight that verifier unreliability can be detected via logical invariants without ground truth, applicable to any verification loop. (3) A practical protocol for integrating this check into existing verification pipelines, including a threshold-based reliability flag.

## Experiment

### Evaluation Setup

| Role | Choice | Rationale (≤12 words) |
| --- | --- | --- |
| Dataset | DafnyBench (500 programs) | Large-scale benchmark with verified Dafny programs |
| Primary metric | Flag accuracy (F1 score) | Measures correct detection of verifier unreliability |
| Baseline 1 | Direct verifier | No consistency check, always assume reliable |
| Baseline 2 | Repetition consistency | Rerun verifier 5 times, flag if inconsistent outputs |
| Baseline 3 | Metamorphic testing (input perturbation) | Perturb code (e.g., rename variables), check verifier consistency |
| Ablation-of-ours | SpecMon (no threshold, no Z3 filter) | Require all spec pairs monotonic, no Z3 verification |
| Additional | Human-annotated spec pairs (100 pairs) | Validate LLM-generated pairs with manual logical order check |
| Preliminary study | Verifier error rate on DafnyBench | Measure how often Dafny verifier gives wrong result (manual inspect 50 cases) |

### Why this setup validates the claim
This combination directly tests the core claim that monotonicity of verifier results across logically ordered specifications reveals verifier unreliability. DafnyBench provides many verification tasks where ground-truth verifier correctness can be established via manual inspection or curated bugs. The direct verifier baseline tests whether any check is needed. Repetition consistency tests whether simple repeated verification suffices; if our method outperforms it, the added value of logical ordering is confirmed. Metamorphic testing isolates the value of logical ordering by using a different consistency principle (input perturbation) rather than specification monotonicity. The ablation removes both the threshold and the Z3 filter, testing whether tolerance of occasional LLM misordering and the Z3 guarantee are essential. The human-annotated set allows us to measure the misorder rate of LLM-generated spec pairs, grounding the assumption that Z3 verification is needed. The preliminary study on verifier error rates motivates the significance of the problem. The primary metric, flag accuracy (F1), directly assesses how well each method identifies verifier unreliability, making the experimental design a falsifiable test of the predictive claim.

### Expected outcome and causal chain
**vs. Direct verifier** — On a case where the verifier incorrectly rejects a valid spec (e.g., due to incompleteness), the baseline always outputs "reliable" because it has no check. Our method generates spec pairs, verifies implication with Z3, and detects monotonicity violation (e.g., strong spec fails but weak spec passes), so it flags "unreliable". We expect the baseline to have 0% detection of such errors, while our method detects >80% of them.

**vs. Repetition consistency** — On a case where the verifier is systematically wrong (identical error each repetition), baseline sees consistent rejects so flags "reliable" (false negative). Our method uses logically ordered spec pairs; even if the verifier is consistently wrong, monotonicity may be violated (e.g., strong reject, weak accept), so it flags reliably. We expect repetition consistency to detect <10% of systematic errors, while our method detects >70%.

**vs. Metamorphic testing** — On a case where the verifier is inconsistent across input perturbations (e.g., code with renamed variables gives different results), metamorphic testing will flag it. However, our method targets spec-level consistency; if the verifier is consistent across code perturbations but fails on spec monotonicity, metamorphic testing gives false negative. We expect metamorphic testing to detect <50% of verifier unreliability cases that involve specification errors, while our method detects >80%.

**vs. SpecMon (no threshold, no Z3 filter)** — On a case where the LLM generates a misordered spec pair (logical implication does not hold), the ablation flags "unreliable" due to that single false inconsistency. Our method with Z3 filter discards such pairs, and with threshold (tau=0.5) tolerates occasional remaining errors. We expect the ablation to have a 40% false alarm rate on reliable verifier cases (due to LLM errors), while our method has <5%.

### What would falsify this idea
If our method's flag accuracy does not significantly exceed the metamorphic testing baseline (e.g., improvement <10% in F1), or if the gain is uniform across all subsets rather than concentrated on cases where systematic verifier errors occur (e.g., incomplete verifier), then the central claim that monotonicity specifically detects verifier unreliability is not supported.
