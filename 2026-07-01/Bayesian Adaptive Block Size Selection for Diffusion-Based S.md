# Bayesian Adaptive Block Size Selection for Diffusion-Based Speculative Decoding

## Motivation

Existing instance-adaptive block size policies (e.g., BlockPilot) require extensive training data covering diverse block sizes to learn a predictor, creating a data-efficiency bottleneck that limits deployment in low-resource or dynamically changing domains. This reliance on pre-collected data stems from the assumption that the relationship between sample features and optimal block size can only be approximated via supervised or reinforcement learning. We identify that the acceptance rate of draft tokens decays geometrically across steps within a block, a structural property that enables closed-form Bayesian inference from a handful of observed accept/reject outcomes without any offline training.

## Key Insight

The acceptance rate of draft tokens follows a geometric distribution with a single parameter (mean acceptance rate), enabling closed-form Bayesian posterior updates from a few observed token outcomes using a conjugate Beta prior.

## Method

We propose BABSS (Bayesian Adaptive Block Size Selection), a method that selects the block size for diffusion-based speculative decoding on a per-instance basis using Bayesian inference on acceptance rates, requiring no training data. Its input is the starting block (first block of tokens generated by the drafter) and its output is the next block size to use.

Assumption: The per-token acceptance rate is constant across positions within a block, implying geometric decay. We verify this assumption empirically by measuring the variance of acceptance rates across positions on a held-out calibration set of 100 prompts from the target domain. If variance exceeds 0.1, we flag potential mismatch, but we proceed with the model as it may still be a useful approximation.

### (B) How it works
The core operation follows a Bayesian update and decision procedure:
```pseudocode
Input: target model T, drafter D, initial block size L_init=4, prior parameters α=1, β=1, C_draft=0.2ms, C_verify=1ms, L_max=16
1. Generate first block of L_init draft tokens using D.
2. Verify each draft token with T, obtaining an accept/reject sequence a_1,...,a_{L_init} (accept=1, reject=0).
3. Update posterior: α' = α + Σ a_i, β' = β + L_init - Σ a_i.
   # Posterior mean = α'/(α'+β') is the MAP estimate of per-token acceptance rate.
4. For candidate block sizes L in [1, L_max]:
   a. Compute expected number of accepted tokens under geometric decay: E_accepted(L) = Σ_{k=1}^L (posterior_mean)^k.
   b. Compute computational cost: C(L) = L * C_draft + L * C_verify.
   c. Compute efficiency: E(L) = E_accepted(L) / C(L).
5. Select L* = argmax_L E(L).
6. Use L* for next block; generate new block with D, verify, and continue.
```
Hyperparameters: prior α=β=1 (uniform), C_draft=0.2ms, C_verify=1ms, L_init=4, L_max=16.

### (C) Why this design
We chose Beta-Binomial over alternative distributions (e.g., exponential family) because the Beta prior is conjugate to the geometric likelihood derived from token-level accept/reject events, enabling exact closed-form updates without sampling. We chose to observe only the first block (L_init=4) rather than multiple blocks to minimize latency, accepting the cost that the posterior may be noisy with very few observations; however, the uniform prior provides regularization. We chose to maximize E_accepted/C(L) instead of using a fixed acceptance probability threshold because it directly accounts for the trade-off between parallelism and rejection cost; the trade-off is that computing the geometric sum E_accepted(L) assumes a constant per-token acceptance rate, which may be violated if the rate changes sharply within the block (e.g., due to long-range dependencies).

### (D) Why it measures what we claim
The quantity `posterior_mean = (α+Σa_i)/(α+β+L_init)` measures the _per-token acceptance rate_ under the assumption that the true acceptance rate is constant across positions within a block (geometric decay with fixed parameter). This assumption fails when draft tokens have position-dependent acceptance rates (e.g., early tokens are more likely accepted than late tokens beyond the geometric trend), in which case `posterior_mean` measures the _average acceptance rate_ over the first block, which may overestimate future acceptance. The expected accepted tokens `E_accepted(L)` measures the _expected number of tokens the verifier will not reject_ under the same constant-rate assumption; if the true acceptance rate decays faster than geometric (e.g., super-exponential), this quantity overestimates actual accepted tokens, leading to suboptimal block sizes. The efficiency ratio `E_accepted/C(L)` measures _computational throughput_ under the model selection criterion; its validity depends on the assumption that computational costs are additive and independent of block size, which holds in our setting because D and T run in constant time per token regardless of block length.

**Verification of assumption:** We test the constant-rate assumption on a held-out calibration set of 100 prompts. For each prompt, we compute the acceptance rate at each position across multiple blocks and measure the variance. If the average variance across positions is ≤0.01, we consider the assumption valid; otherwise, we note potential limitation.

## Contribution

(1) A training-free, few-shot adaptive block size selection algorithm that uses Bayesian inference on token-level accept/reject outcomes from the first block to update a conjugate prior and select subsequent block sizes. (2) Identification of the geometric decay property of draft token acceptance in diffusion speculative decoding as a structural invariant that enables closed-form posterior computation without offline training. (3) A principled efficiency objective that balances expected accepted tokens and computational cost, grounded in the posterior predictive distribution.

## Experiment

### Evaluation Setup

| Role | Choice | Rationale (≤12 words) |
|---|---|---|
| Dataset | CNN/DM, WMT14, ROCStories, MIMIC-III notes (domain shift) | Diverse tasks; MIMIC-III tests OOD robustness. |
| Primary metric | Tokens per second (inference speed) | Measures efficiency gain directly. |
| Baseline | Fixed block size (L=4) | Standard speculative decoding baseline. |
| Baseline | Fixed block size (L=8) | Tests larger fixed block performance. |
| Baseline | BlockPilot (adaptive policy) | State-of-the-art adaptive baseline. |
| Ablation-of-ours | BABSS without posterior update (prior only) | Isolates effect of Bayesian adaptation. |
| Resource | Single NVIDIA RTX 3090 GPU | Feasibility for a consumer GPU. |

### Why this setup validates the claim
The setup tests the central claim—that instance-adaptive block size improves efficiency—by comparing BABSS to fixed small and large block sizes, and to a learned adaptive method. The dataset covers diverse acceptance rate profiles (e.g., high acceptance in translation, low in story generation) and includes a domain shift scenario (MIMIC-III medical text) where BlockPilot may be out-of-distribution. The metric (throughput) directly captures efficiency. Comparing to fixed baselines tests if adaptation beats static choices; comparing to BlockPilot tests if Bayesian online learning matches or exceeds a trained policy. The ablation (no posterior update) isolates the role of observed acceptance, showing whether adaptation is necessary or prior alone suffices. If BABSS outperforms both fixed baselines in their weak regimes and matches BlockPilot on in-distribution data while outperforming on OOD, the claim is validated. Conversely, if gains are uniform or absent, the idea is falsified.

### Expected outcome and causal chain

**vs. Fixed block size (L=4)** — On a prompt with high drafter acceptance rate (e.g., repetitive text), L=4 underutilizes parallelism, causing lower throughput because it always uses small blocks. Our method adaptively selects larger blocks (e.g., L=16) by observing high acceptance, increasing parallelism. We expect a noticeable speedup on high-acceptance prompts but parity on low-acceptance ones.

**vs. Fixed block size (L=8)** — On a prompt with low acceptance rate (e.g., complex reasoning), L=8 wastes computation on many rejections because it commits to a large block. Our method shrinks block size adaptively, reducing wasted tokens. We expect faster generation on low-acceptance prompts but similar speed on high-acceptance ones.

**vs. BlockPilot** — On in-distribution prompts (CNN/DM, WMT14, ROCStories), BlockPilot is expected to perform well due to training on similar data; BABSS should be comparable. On the out-of-distribution MIMIC-III domain, BlockPilot may select suboptimal block sizes due to distribution shift, while BABSS adapts online. We therefore expect BABSS to outperform BlockPilot on MIMIC-III, validating robustness without training.

**Ablation (no posterior update)** — If BABSS without posterior update (using prior mean 0.5) achieves similar throughput to full BABSS, then adaptation is not beneficial; the claim of instance-adaptivity would be unsupported.

### What would falsify this idea
If BABSS fails to outperform the best fixed block size on both high- and low-acceptance prompts, or if the ablation (without posterior update) matches BABSS, then the central claim of adaptive block size is unsupported. Additionally, if on MIMIC-III, BlockPilot still matches or beats BABSS, then the domain shift advantage is not realized.
